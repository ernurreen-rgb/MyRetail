# Contributing

## Branches

- `main` must remain buildable.
- Create one short-lived branch per task.
- Use descriptive names such as `feat/product-list` or `fix/tenant-routing`.

## Pull requests

- Explain the user-visible or technical outcome.
- Link the related task and Notion specification.
- Include tests for changed behavior.
- Request review from the other implementation team member.
- Do not merge while CI is failing.

## Definition of done

- Acceptance criteria are satisfied.
- Lint, type checks, tests, and production build pass.
- Security and tenant-isolation implications were reviewed.
- Operational or architectural changes are documented in Notion.
